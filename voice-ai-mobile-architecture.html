Jitter Buffer — Playback Smoothing

Streaming TTS arrives in variable-size chunks over WebSocket. Without buffering, audio stutters when network jitter causes late arrivals. The 200ms jitter buffer pre-fills before initiating playback.

Buffering Playing ■ Empty slot
200ms = ~5 chunks @ 100ms intervals before playback starts

Barge-In Logic — Step by Step
1
Continuous VAD During Playback
VAD runs on microphone input at all times — even while AI audio is playing. AEC removes speaker signal before VAD analysis.
2
VAD Threshold Crossed
Energy + zero-crossing rate exceeds threshold. Minimum 80ms of sustained speech to avoid false positives from coughs or clicks.
3
Immediate Interrupt
Audio playback stops immediately. Playback buffer flushed. AudioQueue reset. barge_in event sent over WebSocket.
4
Resume Listening
Microphone remains active (persistent state). UI transitions: speaking → barge_in → listening. Backend handles LLM interrupt.
Echo Cancellation Architecture
⚠ PROBLEM
Device speaker plays AI audio → microphone picks it up → VAD falsely triggers → spurious barge_in events
⚙ SOLUTION
iOS built-in AEC via AVAudioSession + .voiceChat mode. Android via WebRTC AEC API or java AudioEffect. Subtracts known reference signal.
✓ RESULT
VAD only reacts to user voice. AI speech at full volume on speaker. Clean barge-in with <150ms latency P95.
§ 04

Persistent Microphone State

Mic Lifecycle Management

Typical push-to-talk apps stop and restart the mic every turn — causing 200–800ms dead time. Our approach: mic session opens once at conversation start, stays open for the full session.

Session Init → Mic OPEN
AVAudioSession configured once. AudioEngine starts. Tap installed on input node.
AI Speaking → Mic STAYS OPEN
PCM capture continues. AEC active. VAD monitors. No tap removal.
AI Done → Mic ALREADY READY
No reactivation needed. User speaks immediately. Zero tap gap.
Session End → Mic CLOSED
AudioEngine stops. Session deactivated. Resources released.
Flutter Implementation Snippet
Dart
// Persistent mic — init once, keep alive
class VoiceSessionManager {
  AudioSession? _session;
  bool _micActive = false;

  Future<void> initSession() async {
    _session = await AudioSession.instance;
    await _session!.configure(AudioSessionConfiguration(
      avAudioSessionCategory:
        AVAudioSessionCategory.playAndRecord,
      avAudioSessionMode:
        AVAudioSessionMode.voiceChat, // AEC
    ));
    await _startCapture();
    _micActive = true;
  }

  // Mic stays open — only mute VAD sending
  void onAISpeaking() =>
    _vadSendEnabled = false; // capture on

  void onAIDone() =>
    _vadSendEnabled = true; // ready instantly
}

⚠ AUDIO LEAK SCENARIOS (Must Prevent)
• Stuck in SPEAKING: TTS socket closes before tts_done received → implement explicit timeout fallback
• PROCESSING loop: LLM times out → max 10s timeout with fallback to IDLE
• Mic not released: Background audio session conflict → deactivate on app background
• Buffer not flushed: Barge-in with residual queue → synchronous flush before state change
§ 07

Flutter + Swift Hybrid Architecture

Flutter ↗
Cross-platform layer. Handles UI rendering, state management, WebSocket communication, and orchestration.
  • UI state machine (idle/listen/process/speak)
  • WebSocket connection lifecycle management
  • Audio session configuration via audio_session plugin
  • VAD event processing and barge_in dispatch
  • Chat history and conversation UI
  • Cross-platform Android + iOS build target
Swift ↗
Native iOS audio layer. Low-latency capture and playback via Apple's core audio APIs.
  • AVAudioEngine tap for 16kHz PCM capture
  • AudioQueue for 24kHz PCM playback
  • AVAudioSession .voiceChat mode (AEC)
  • Real-time jitter buffer management
  • Platform channel bridge to Flutter
  • Hardware-level echo cancellation
Why Both Layers?
Flutter Alone — Risk Flutter audio plugins (flutter_sound, record) add buffering overhead. Barge-in latency often 300–500ms. No direct AVAudioEngine access.
Hybrid — Optimal Flutter for UI/state/WS. Swift native for audio I/O. Platform channels pass PCM buffers and events. Best of both worlds.
Android Path Flutter with Oboe via NDK for low-latency capture. WebRTC AEC via AudioEffect API. Same pipeline interface.
